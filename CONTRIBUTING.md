## Code of Conduct

This project follows the [Contributor Covenant Code of Conduct](CODE_OF_CONDUCT.md). By participating, you are expected to uphold this code. Please report unacceptable behavior to the project maintainers.

## How Can I Contribute?

### Reporting a Bug

If you find a bug or a broken link:

1. Check that the issue hasn't been reported before
2. Create a new issue using the [Bug Report](.github/ISSUE_TEMPLATE/bug_report.md) template
3. Provide all relevant information

### Suggesting a New Resource

To add a new resource to the list:

1. Verify that the resource is not already in the list
2. Make sure it meets the [Acceptance Criteria](#acceptance-criteria)
3. Create a Pull Request following the [Pull Request Guide](#pull-request-guide)

### Improving Documentation

Documentation improvements are always welcome:

- Fix typos
- Improve description clarity
- Add new categories
- Improve list organization

### Asking a Question

If you have a question:

1. Search existing issues to see if it has been discussed
2. If you don't find anything, create a new issue using the [Question](.github/ISSUE_TEMPLATE/question.md) template

## Acceptance Criteria

Unfortunately, not every resource can be included. The aim of the list is to provide a concise list of noteworthy modern software. A suggested resource must meet at least one of the following criteria:

**(a) Widely recommended** - Regardless of personal opinion, it is widely recommended by the community.

**(b) Highly discussed** - Due to its innovative nature, it generates active discussion in the community.

**(c) Unique in its approach** - It has an absolutely unique approach and function.

**(d) Niche product** - It fills a specific gap that is not covered by other tools.

**Important note:** Self-promotion is frowned upon and will be reviewed critically, but the suggestion will be approved if it meets the criteria.

## Pull Request Guide

### Before Creating a PR

1. **Check for duplicates**: Make sure the resource is not already in the list
2. **Verify criteria**: Confirm that the resource meets the [Acceptance Criteria](#acceptance-criteria)
3. **Review format**: Follow the [Entry Format](#entry-format)

### Pull Request Process

1. **Fork the repository**
2. **Create a branch** (optional, but recommended):
   ```bash
   git checkout -b add-resource-name
   ```
3. **Make your changes** following the correct format
4. **Commit your changes**:
   ```bash
   git commit -m "Add: Resource Name"
   ```
5. **Push to your fork**:
   ```bash
   git push origin add-resource-name
   ```
6. **Open a Pull Request** on GitHub

## Entry Format

### Basic Format

```
[**Resource Name**](URL) - Brief description of the resource.
```

### Example

```
[**Self.xyz Documentation**](https://self.xyz/docs) - Official Self.xyz documentation.
```

### Format Rules

- **Alphabetical order**: Entries should be sorted in ascending alphabetical order (a-z)
- **First letter uppercase**: The first letter of each entry should always be capitalized
- **Short descriptions**: Keep descriptions brief, simple, and unbiased
- **End with period**: All descriptions must end with a period
- **No trailing whitespace**: Make sure your editor removes trailing whitespace
- **Spelling and grammar**: Check your spelling and grammar before submitting

### Categories

If you want to add a new category or improve existing categorization:

1. Create an issue first to discuss the new category
2. Make sure the category makes sense and is not too specific
3. Add a brief description of the category if necessary

## Reporting Issues

If you find a problem (broken link, typo, etc.):

1. Verify that the issue hasn't been reported before
2. Use the [Bug Report](.github/ISSUE_TEMPLATE/bug_report.md) template
3. Provide:
   - Clear description of the problem
   - Steps to reproduce (if applicable)
   - Expected behavior vs. actual behavior

## Questions

If you have a question about:

- Whether a resource is appropriate for the list
- How to format an entry
- Any other questions about the project

1. Search existing issues to see if it has been discussed
2. If you don't find anything, create a new issue using the [Question](.github/ISSUE_TEMPLATE/question.md) template

## Pull Request Review

Maintainers will review PRs and may:

- Request changes if the format is not correct
- Ask for more information about a resource
- Approve and merge the PR

Please be patient during the review process. All PRs are manually reviewed to maintain the quality of the list.
